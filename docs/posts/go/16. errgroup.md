# 1. 引言

errgroup是go拓展包`golang.org/x/sync`提供的带有context控制机制的并发处理的结构，其不仅可以实现`waitgroup`的功能，还可以具备error处理、context取消及控制goroutine num的功能，非常适合并发逻辑的编写。

# 2. 源码

errgroup的结构核心逻辑如下：

```go
type Group struct {
	cancel func(error)     // 取消函数
	wg     sync.WaitGroup  // 等待组
	sem    chan token      // 信号量，用于限制并发数
	errOnce sync.Once      // 确保错误只被设置一次
	err    error           // 存储第一个错误
}
```

`errOnce`和`err`共同实现了其可以进行error处理的逻辑，errgroup对error的处理逻辑是，捕获第一个发生错误的goroutine，为了防止多goroutine同时对error进行赋值的情况，其采用了单例机制来保证error的赋值，核心逻辑如下：

```go
if err := f(); err != nli {
	g.errOnce.Do(func(){
		g.err = err
		if g.cancel != nil {
			g.cancel(g.err)
		}
	})
}
```

# 3. 核心方法

## 3.1 WithContext

当调用`errgroup.WithContext`方法时，会创建一个新的Group和派生的Context，当第一个goroutine返回非nil错误或Wait返回时，Context会被取消。在捕捉到error后，会同时调用`g.cancel()`方法，该方法在调用 `WithContext(ctx)`方法时进行的注册和赋值，代码逻辑如下：

```go
func WithContext(ctx context.Context) (*Group, context.Context) {  
    ctx, cancel := withCancelCause(ctx)  
    return &Group{cancel: cancel}, ctx  
}
```

这也是为什么`errgroup`可以在捕获error后通知所有其派生的子goroutine退出的原因，其本质是利用了`context`的cancel机制。

## 3.2 Go、Wait、SetLimit、TryGo

`Go`方法提供了启动新的goroutine的能力，是errgroup的核心方法：
- 如果设置了并发限制，会阻塞直到可以添加新的goroutine；
- 第一个返回非nil错误的goroutine会取消Context；
- 错误会被`Wait()`返回；
`Wait`方法阻塞等待所有的goroutine完成，返回第一个非nil错误。
`SetLimit`限制并发goroutine数量，负数表示无限制，传0则会创建一个无缓冲的channel，每次`g.sem <- token{}`都会永久阻塞，其本质使用了`channel`的阻塞原理，其源码如下：

```go
func (g *Group) SetLimit(n int) {
	if n < 0 {
		g.sem = nil
		return
	}
	if active := len(g.sem); active != 0 {
		panic(fmt.Errorf("errgroup: modify limit while %v goroutines in the group are still active", active))
	}
	
	g.sem = make(chan token, n)
}	
```

下的代码片段是使用`Go`方法启动goroutine时的一段逻辑：
```go
if g.sem != nil {
  g.sem <- token{}
}
```

所以，假如sem现在是满的，代码就会阻塞在这里，等待有其他goroutine完成任务退出后才能继续执行逻辑。

`TryGo`的语义是非阻塞地尝试启动 goroutine，若并发槽已满则直接返回 `false`，不会阻塞调用方，适用于不想让任务入队等待、而是直接丢弃的场景。

# 4. 使用场景

## 4.1 并行聚合

这是`errgroup`最常见的使用场景，假如某个复杂逻辑需要整合几个不同的下游服务共同完成，任何一个服务报错，整个请求就应该立即失败，并取消其他还在执行的下游请求，以节省系统资源。示例代码如下：

```go
func GatewayAggregate(ctx context.Context, userId string) error {  
    eg, egCtx := errgroup.WithContext(ctx)  
  
    // 查数据库  
    eg.Go(func() error {  
       return fetchUserFromDB(egCtx, userId)  
    })  
  
    // 外部连接调用  
    eg.Go(func() error {  
       return fetchBalanceFromRPC(egCtx, userId)  
    })  
  
    // 拉取用户头像，非关键资源，失败后可降级 
    eg.Go(func() error {  
       _= fetchAvatar(egCtx, userId)  
		
       return nil  
    })  
  
    err := eg.Wait()  
    if err != nil {  
       return fmt.Errorf("eg.Wait(): %w", err)  
    }  
  
    return nil  
}

```

上述`GatewayAggregate`演示了利用`errgroup`并行处理任务，并在某个服务失败后利用fail-fast的模式快速结束逻辑。如果传入原始 `ctx` 而非 `egCtx`，当某个 goroutine 报错触发 cancel 时，其他 goroutine 监听的是原始 `ctx`，不会感知到取消信号，从而继续执行直到自然结束，`Wait()` 也需要等待它们全部完成。

## 4.2 控制并发量

例如存在某个场景是获取区块链数据并写入数据库，如果不加限制地直接 `go func`，瞬间拉起上千个协程可能会把下游数据库或其他中间件的连接池打满。这里可以利用 `SetLimit`方法，控制并发完成任务。

```go
func BatchProcessBlocks(ctx context.Context, blockHeights []int64) error {  
    eg, egCtx := errgroup.WithContext(ctx)  
  
    eg.SetLimit(20)  
  
    for _, height := range blockHeights {  
	    h := height  
  
       // 如果 eg 内部的并发数已经达到 20，这里的 eg.Go 会被阻塞，直到有任务完成  
       eg.Go(func() error {  
          // 每次执行前检查一下 context 是否已经被取消，避免无谓的计算  
          if egCtx.Err() != nil {  
             return egCtx.Err()  
          }  
  
          return processSingleBlock(egCtx, h)  
       })  
    }  
  
    // 等待所有批次处理完成  
    if err := eg.Wait(); err != nil {  
       return fmt.Errorf("批量处理中断: %w", err)  
    }  
  
    return nil  
}
```

注意，我们在goroutine内部处理逻辑中增加了一段 `egCtx.Err() != nil`的判断逻辑，目的是防止某些情况下，errgroup已经收到了退出信号并进行退出了，但新到来的goroutine在整体eg发出`cancel`方法时是处于阻塞状态，所以其仍会继续执行任务，造成资源浪费，为了避免该情况，应该主动在执行任务前检测主体errgroup是否已经取消了任务。

## 4.3 goroutine panic

这其实是所有利用goroutine进行异步任务逻辑时都需要注意的问题，结合`errgroup`梳理下如何编写panic safe的代码。

```go
func SafeGo(eg *errgroup.Group, task func() error) {  
    eg.Go(func() (err error) {  
       // 拦截 panic 并转化为 error       
       defer func() {  
          if r := recover(); r != nil {  
             err = fmt.Errorf("panic in errgroup: %v\n", r)  
          }  
       }()  
  
       return task()  
    })  
}
```

其实就是在panic的`recover`中对`recover`的返回值进行处理，使其可以抛出这个错误，防止errgroup无法感知panic的发生。
